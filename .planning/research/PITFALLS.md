# Domain Pitfalls

**Domain**: A 股量化选股系统 (Event Scorecard-Driven Stock Selection with LLM)
**Researched**: 2026-04-19
**Confidence**: HIGH (verified with official quant research sources + domain-specific post-mortems)

---

## Critical Pitfalls

### Pitfall 1: 回测过拟合 (Backtest Overfitting)

**What goes wrong**: 
回测曲线完美 (年化 30%+, 最大回撤<10%),实盘持续亏损。系统有大量自由参数 (阈值、权重、衰减系数),通过参数挖掘找到了历史最优组合，但对未来无预测能力。

**Why it happens**: 
- A 股历史短、制度变迁频繁 (注册制、涨跌停改革、融券新规),历史规律易失效
- 开发者看到回测不佳就调参数，直到曲线"好看"为止
- 因子/参数数量远超样本量承载能力 (如 1200 个交易日数据用 20 个参数)

**Consequences**: 
- 实盘表现与回测差异巨大 (年化收益从 30% 降至 -10%)
- 策略上线 3-6 个月后 alpha 迅速衰减
- 用户信任崩塌，系统被弃用

**Prevention**: 
- **参数数量限制**: 参数≤√样本量 (如 5 年 1200 日数据，参数≤35 个)
- **滚动窗口验证**: 用过去 N 年训练，接下来 M 个月验证，滚动至少 3 轮完整牛熊周期
- **样本外封存**: 至少保留最近 6 个月数据完全不用，用于最终验证
- **敏感性测试**: 关键参数上下浮动 20%,策略表现不应剧烈变化
- **本项目应用**: REQUIREMENTS.md 中所有阈值 (0.3/0.5/0.75、重要性锚点 0.9/0.7/0.3) 必须标注为"待回测验证"

**Warning signs**: 
- 回测报告没有样本外验证章节
- 参数调整记录不完整 ("试了很多组，这组最好"是危险信号)
- 策略在 2024 年 2 月小盘股流动性危机等极端行情中表现异常好 (可能过拟合特定行情)

**Phase to address**: Phase 0.5 设计阶段即需预留回测接口，所有参数配置化

### Pitfall 2: 未来函数 (Look-Ahead Bias)

**What goes wrong**: 
回测中使用了"当时还不知道"的信息，导致回测虚高，实盘无法复现。

**Why it happens**: 
- 财报发布日期未建模：假设财报发布当日即可交易，但实际 A 股财报多在收盘后发布，最早次日才能交易
- 宏观数据对齐错误：PMI 每月最后一天发布、M1/M2 次月 10-15 日发布，回测假设当月 1 日就能拿到
- 涨跌停处理不当：回测假设"触发信号当日涨停仍可买入"，实际一字板根本买不到
- 新闻时间戳不精确：假设新闻发布时刻就能拿到并计算信号，实际爬虫有延迟

**Consequences**: 
- 回测年化收益虚高 5%-15%
- 实盘无法复现回测结果
- 关键参数 (如止损阈值) 基于错误回测设定，实盘风控失效

**Prevention**: 
- **统一交易日历**: 全系统使用 A 股交易日历，所有日期计算基于交易日
- **数据发布延迟建模**: 
  - 财报：T 日收盘后发布→最早 T+1 日可交易
  - 宏观数据：根据历史发布日历建模 (如 REQUIREMENTS.md 中 L0 层的 M1-M2_PENDING 状态)
  - 新闻：假设 T+0 日可获取但需考虑爬虫延迟
- **涨跌停可交易性检查**: 回测中涨停=买不到，次日再判断；跌停=卖不掉，复牌才能操作
- **收盘价替代**: 使用 14:55 价格近似收盘价，避免"收盘瞬间计算信号"的假设

**Warning signs**: 
- 回测框架未考虑财报发布日期，直接用财报季度切分
- 信号计算假设"收盘后立即执行"，但实际数据 15:05 后才完整
- 回测未模拟涨跌停约束

**Phase to address**: Phase 0.5 基础设施层 (INFRA-01/02/03)

### Pitfall 3: LLM 直接生成交易信号

**What goes wrong**: 
让 LLM 阅读新闻后直接输出"买入/卖出"建议，实盘发现 LLM 对"利好/利空"的判断与市场实际反应经常相反。

**Why it happens**: 
- LLM 没有经过金融领域专业微调，不理解"利好出尽是利空"等市场博弈逻辑
- LLM 存在幻觉，可能编造不存在的公告或数据
- LLM 知识截止日期滞后，无法获知最新市场共识
- 同一新闻让 LLM 跑 10 次，可能输出 7 种不同的强度分数 (0.6-0.9 波动)

**Consequences**: 
- 信号一致性差，同一输入不同时间输出不同
- LLM 错误归因导致系统推荐错误标的
- 合规风险：LLM 直接输出交易指令可能触碰监管红线

**Prevention**: 
- **LLM 职责边界**: LLM 只做信息提取 (事件类型、涉及公司、利好方向定性)、逻辑分类 (产业趋势/政策驱动/事件驱动/流动性驱动)，不做强度打分
- **Skill 锚定等级分类**: LLM 根据锚点描述归类到"高/中/低"等级，由规则引擎映射为具体数值 (如高→0.7)
- **置信度过滤**: LLM 输出附带置信度，低于阈值的结果丢弃或人工复核
- **规则兜底**: LLM 失败时降级为关键词匹配或沿用上一日数据 (如 REQUIREMENTS.md 中的 LLM 降级策略)
- **本项目应用**: REQUIREMENTS.md 中 LLM 输出契约 (4.2 节) 设计正确——LLM 不输出强度分数

**Warning signs**: 
- Prompt 中包含"请给出 0-1 的强度评分"
- 无置信度字段，所有输出同等对待
- 无降级方案，LLM 挂掉系统就停摆

**Phase to address**: Phase 0.5 L1 层 (LOGIC-01/02)

### Pitfall 4: 逻辑与价格因果倒置

**What goes wrong**: 
用市场涨幅反推逻辑强度——板块涨了就说逻辑强，跌了就说逻辑弱，丧失了逻辑层的独立判断价值。

**Why it happens**: 
- 逻辑强度计算中混入价格数据 (如用近 5 日涨幅修正 logic_strength)
- LLM 阅读包含价格信息的新闻 ("XX 板块大涨 5%,原因是...")
- 设计时未严格隔离逻辑层 (L1) 与市场层 (L2) 的数据流

**Consequences**: 
- 系统变成"追涨杀跌"的动量策略，丧失了"逻辑独立于价格"的核心价值
- 无法在"逻辑已强化但市场未反应"的左侧机会出现时发出信号
- 市场非理性下跌时系统错误否定逻辑，错失反弹机会

**Prevention**: 
- **数据隔离铁律**: L1 绝不读取 L2/L3 的任何价格数据；L2 绝不读取 L1 的逻辑证据
- **逻辑强度纯事件驱动**: 仅基于政策发布、业绩公告、产业数据等非价格事件计算
- **市场认知纯量价驱动**: 仅基于均线排列、涨停家数、换手率等量价数据计算
- **输出对比信号**: 当价格大跌而逻辑层无负面事件时，系统应输出"逻辑层面无变化，市场层面出现抛压，建议观察支撑位"

**Warning signs**: 
- logic_strength 计算公式中包含 price_change 或 relative_strength
- LLM Prompt 中包含板块近 N 日涨幅信息
- 逻辑层读取了板块指数行情数据

**Phase to address**: Phase 0.5 架构设计阶段 (总体架构数据隔离铁律)

### Pitfall 5: 数据源不可用导致系统停摆

**What goes wrong**: 
假设某个数据源 (如北向资金、涨停板数据、LLM API) 永远稳定，实际遇到爬虫反制、API 限流、服务宕机时系统完全停摆。

**Why it happens**: 
- 单点依赖：系统只接入了一个数据源 (如只用 Tushare 获取涨停数据)
- 无降级设计：假设所有模块都完美运行，未考虑 fallback 逻辑
- 延迟容忍度不足：假设收盘瞬间就能拿到所有数据并计算信号

**Consequences**: 
- 关键交易日 (如市场大跌日) 系统无法输出推荐，错失机会
- 用户看到系统输出"N/A"或空列表，信任度下降
- 实盘仓位被迫降低 (因为信号不完整)

**Prevention**: 
- **数据源冗余**: 同时接入多个数据源 (如涨停数据：Tushare + 东方财富爬虫),一个挂掉自动切换
- **完整降级链路** (如 REQUIREMENTS.md 第 9 节):
  - LLM 服务不可用→沿用上一日逻辑 + 自然衰减
  - 涨停板数据缺失→情绪面仅用换手率，过热判定阈值从 1.5 提高到 2.0
  - 北向资金缺失→资金面不计入，综合分按技术 + 情绪二维计算
  - 宏观数据连续两月不可用→macro_multiplier=1.00
- **延迟容忍度测试**: 模拟"晚 30 分钟拿到数据"对结果的影响
- **健康检查**: 每日运行前检查各数据源状态，核心数据源失效时告警并中止

**Warning signs**: 
- 代码中无 try-catch 或 fallback 逻辑
- 配置文件只有一个数据源 URL
- 无数据源健康监控仪表盘

**Phase to address**: Phase 0.5 基础设施层 (INFRA-04 日志与监控)

### Pitfall 6: A 股特有交易约束未建模

**What goes wrong**: 
回测假设可以无限资金按收盘价成交，实盘发现小盘股根本买不进/卖不出，或冲击成本极高。

**Why it happens**: 
- 未考虑涨跌停约束：涨停买不到，跌停卖不掉
- 未考虑停牌约束：停牌期间无法交易，复牌可能连续跌停
- 未考虑冲击成本：小盘股日均成交仅几千万，500 万订单就能推高价格 3%-5%
- 未考虑 T+1 约束：当日买入次日才能卖出

**Consequences**: 
- 回测年化收益 30%,实盘仅 10%(冲击成本吃掉 alpha)
- 关键交易日 (如信号触发当日涨停) 无法建仓，错过整个波段
- 持仓个股跌停时无法止损，回撤远超预期

**Prevention**: 
- **涨跌停可交易性检查** (如 REQUIREMENTS.md 7.2 节):
  - 当日涨停→不推荐买入，标记"观察，等待分歧"
  - 当日跌幅>7%→仓位降一档，标记"追高风险"
- **流动性过滤**: 近 20 日日均成交<5000 万的标的，即使信号再强也不推荐或极小仓位
- **冲击成本回测**: 回测中扣除 0.1%-0.3% 的滑点 (小盘股更高)
- **T+1 建模**: 回测中当日买入的信号，次日才能卖出

**Warning signs**: 
- 回测报告未提及涨跌停处理
- 推荐列表包含日均成交<1 亿的标的
- 止损规则假设"收盘价跌破即卖出",未考虑跌停卖不掉

**Phase to address**: Phase 0.5 L4 层 (EXEC-02 A 股交易约束检查)

### Pitfall 7: 宏观敏感度矩阵维护成本爆炸

**What goes wrong**: 
设计几百个板块×5 个宏观维度=上千个敏感度参数，维护成本极高且无法适应风格切换。

**Why it happens**: 
- 试图为每个板块预设静态敏感度 (如"银行对流动性敏感度=0.7")
- 未考虑同一板块内不同逻辑敏感度不同 (如半导体板块的"国产替代"逻辑 vs"周期复苏"逻辑)
- 风格切换时原参数失效 (如 2023 年价值风格→2024 年成长风格)

**Consequences**: 
- 参数维护成为巨大负担，每新增一个板块就要设计 5 个参数
- 参数与实际敏感度偏离，宏观层输出失真
- 风格切换时系统推荐持续错误

**Prevention**: 
- **逻辑族映射替代板块映射** (如 REQUIREMENTS.md 3.4 节):
  - 产业趋势逻辑→对"增长动能"敏感 (+0.7)
  - 流动性驱动逻辑→对"流动性环境"敏感 (+0.9)
  - 政策驱动逻辑→对"政策"敏感 (+0.8)
  - 仅需维护约 20 个参数 (逻辑族×维度)
- **滚动回归动态计算**: 用历史数据计算各逻辑族对宏观指标的相关性，季度更新
- **Phase 0.5 简化**: 宏观仅作为全局乘数 (macro_multiplier),不做板块差异化

**Warning signs**: 
- 配置文件中存在 sector_macro_sensitivity 矩阵且超过 50 行
- 新增板块时需要手动设置 5 个宏观敏感度参数
- 宏观层输出与板块实际表现相关性低

**Phase to address**: Phase 0.5 L0 层 (宏观环境层设计)

## Moderate Pitfalls

### Pitfall 8: 分歧系数参数未经验证

**What goes wrong**: 
设计了复杂的分歧系数公式 (divergence = 2*min(pos,neg)/max(pos,neg)) 和阈值 (>0.7 高分歧),但从未回测验证。

**Why it happens**: 
- 参数来自"拍脑袋"而非数据验证
- 公式设计过于复杂，难以理解经济含义
- 未做敏感性测试 (阈值从 0.7 改为 0.6 或 0.8 表现如何?)

**Prevention**: 
- **Phase 0.5 简化**: 仅标记"有无正反逻辑"(如 REQUIREMENTS.md 4.4.1 节 has_headwind)
- **回测验证后再引入**: 积累足够数据后，用回测确定最优阈值
- **参数可配置**: 所有阈值放在独立配置文件，便于调优

**Phase to address**: Phase 0.5 简化为 has_headwind 标记，Phase 2 再引入分歧系数

### Pitfall 9: 中军定义跨板块不可比

**What goes wrong**: 
用"市值前 5 名"定义中军，在大盘股板块 (如医药，前 5 市值>500 亿) 和小盘股板块 (如某个细分概念，前 5 市值<50 亿) 含义完全不同。

**Why it happens**: 
- 用绝对排名替代相对地位
- 未考虑板块内市值分布差异

**Prevention**: 
- **板块内百分位定义** (如 REQUIREMENTS.md 6.5.3 节):
  - 中军 = 市值>75 百分位 + 成交额>75 百分位 + 近 60 日涨跌幅不在后 25%
- **涨幅过滤**: 排除长期下跌的"前中军"(地位仍在但趋势已坏)

**Phase to address**: Phase 0.5 L3 层 (STOCK-07 龙头/中军/跟风识别)

### Pitfall 10: 新闻统计滞后于实际逻辑强度

**What goes wrong**: 
用新闻/研报数量统计作为逻辑强度指标，实际发现新闻密集时板块已上涨 30%+,信号严重滞后。

**Why it happens**: 
- 新闻/研报通常在逻辑发酵后才会密集报道
- 机构可能配合出货故意放消息
- 板块间新闻密度不可比 (热门板块天生新闻多)

**Prevention**: 
- **事件计分板规则引擎**: 基于结构化事件 (政策发布、业绩公告、产业数据) 计算，而非新闻数量
- **逻辑等级分类**: LLM 输出"高/中/低"等级而非连续分数，降低噪音敏感度
- **逻辑被追踪天数**: 用"该逻辑被系统追踪的天数"判断新逻辑 vs 已确认逻辑

**Phase to address**: Phase 0.5 L1 层 (事件计分板设计)

## Minor Pitfalls

### Pitfall 11: 复权数据处理不当

**What goes wrong**: 
回测使用前复权，实盘使用不复权，导致技术指标计算结果不同。

**Prevention**: 
- 全链路统一使用后复权，仅在最终下单时转回不复权
- 每日更新复权因子表

**Phase to address**: Phase 0.5 INFRA-01 数据库设计

---

### Pitfall 12: 用户手动维护成本过高

**What goes wrong**: 
要求用户每天手动更新龙头列表、逻辑评分，实际用户不会坚持。

**Prevention**: 
- 自动识别为主 (规则判定龙头/中军)
- 人工覆盖为辅 (仅提供纠错机制，非日常维护)

**Phase to address**: Phase 0.5 WEB-03 手动覆盖功能 (定位为纠错而非日常)

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| **硬编码阈值** (如 0.3/0.5/0.75) | 快速上线 | 无法回测调优，过拟合风险高 | **Never** — 必须配置化 |
| **单一数据源** | 开发简单 | 数据源挂掉系统停摆 | Phase 0.5 MVP 可接受，但需有降级方案 |
| **LLM 自由打分** | Prompt 简单 | 输出不稳定，实盘不可靠 | **Never** — 必须 Skill 锚定 |
| **无回测验证** | 节省时间 | 参数基于猜测，实盘失效概率高 | **Never** — 至少做样本外验证 |
| **混入盘中数据** | 信号更及时 | 回测与实盘不一致，未来函数 | **Never** — 只用收盘后可获取数据 |
| **手工标注历史数据** | 可立即回测 | 标注成本高，难以扩展 | Phase 0.5 可接受 (3-5 个典型板块),Phase 2 需自动化 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| **Tushare API** | 假设接口永远稳定，无限流限 | 设置请求频率限制，本地缓存每日数据，token 失效时告警 |
| **LLM API** | 假设输出格式永远正确 | JSON 解析失败时重试一次，仍失败则降级为上一日数据 |
| **东方财富爬虫** | 高频爬取触发反爬 | 设置合理爬取间隔 (如 1 秒/请求),使用代理池，准备备选数据源 |
| **MySQL 数据库** | 未建索引导致查询慢 | events 表按 (logic_id, expire_date) 建复合索引，每日全量重算时性能关键 |
| **定时任务调度** | 假设每日计算能在 1 小时内完成 | 设置超时告警，计算超时则输出降级结果 (如仅市场层驱动) |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| **每日全量重算事件表** | 计算时间随事件量线性增长 | 设置事件归档策略 (expire_date<30 日前移入 archive 表) | 事件量>1000 条/日时开始变慢 |
| **LLM 批量调用** | API 费用爆炸，响应时间>5 分钟 | 缓存新板块关键词库，产业链定位缓存季度刷新 | 板块数>100 或每日新闻>500 条时 |
| **复杂 SQL 联表查询** | 推荐列表生成>30 秒 | 预计算中间表 (如个股-板块暴露表),每日增量更新 | 股票池>500 只或板块数>200 个时 |
| **无限制回测数据加载** | 回测 OOM 崩溃 | 分批加载数据，使用生成器而非一次性加载全量 | 回测周期>5 年或股票池>1000 只时 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| **API Token 硬编码** | Token 泄露导致额度被盗用 | 使用环境变量或加密配置文件，Git 忽略敏感文件 |
| **用户推荐列表公开** | 被他人利用进行"抬轿子"操作 | 推荐列表仅用户本人可见，API 需鉴权 |
| **手动覆盖无审计** | 恶意篡改推荐结果 | 记录所有人工覆盖操作 (时间、用户、修改内容) |
| **数据库无备份** | 数据丢失导致系统瘫痪 | 每日自动备份 events、logics 等核心表 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| **推荐列表无解释** | 用户不知道为何推荐这只股 | 每只股附带逻辑摘要、市场强度分、催化剂标记 |
| **仓位建议太模糊** | 用户不知道"半仓"是 20% 还是 50% | 明确标注"建议单票仓位 20-30%" |
| **无止损条件** | 用户不知道何时卖出 | 明确标注"收盘跌破 20 日线止损" |
| **宏观层信息缺失** | 用户不知道当前宏观环境 | 每日输出宏观四象限判定结果与 macro_multiplier |

## "Looks Done But Isn't" Checklist

- [ ] **回测验证**: 所有阈值参数是否有样本外验证报告? — 验证方法：查看回测报告中是否有滚动窗口验证结果
- [ ] **数据源降级**: 每个数据源是否有 fallback 逻辑? — 验证方法：模拟 Tushare API 超时，观察系统是否降级运行
- [ ] **涨跌停约束**: 回测是否正确模拟涨跌停不可交易? — 验证方法：检查回测代码中是否有涨停买入过滤逻辑
- [ ] **LLM 输出契约**: LLM 是否只输出定性分类而非定量分数? — 验证方法：查看 Prompt 中是否包含"请输出 0-1 分数"
- [ ] **数据隔离**: 逻辑层是否真的不读取价格数据? — 验证方法：代码审查 L1 层是否 import 了行情数据
- [ ] **复权统一**: 回测与实盘是否使用同一复权方式? — 验证方法：对比回测与实盘的 MA20 计算结果
- [ ] **时间戳对齐**: 财报发布日期是否正确建模? — 验证方法：检查财报数据是否使用了正确的发布延迟

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| **过拟合发现** | HIGH | 1. 立即暂停策略 2. 减少参数数量 3. 重新做滚动窗口验证 4. 小资金重新实盘测试 |
| **数据源永久失效** | MEDIUM | 1. 切换到备选数据源 2. 验证新数据与旧数据一致性 3. 必要时重新校准阈值 |
| **LLM 输出持续错误** | MEDIUM | 1. 降级为规则匹配 2. 收集错误样本优化 Prompt 3. 考虑切换 LLM 供应商 |
| **实盘大幅偏离回测** | HIGH | 1. 暂停策略 2. 对比回测与实盘的数据/计算逻辑差异 3. 修复后纸面交易验证 4. 小资金重新测试 |
| **宏观判断连续错误** | LOW | 1. 将 macro_multiplier 固定为 1.00 2. 分析错误原因 (数据延迟？四象限规则问题？) 3. 校准后重新启用 |
