## 0. 可用数据源全景

先搞清楚我们手里有什么牌，再决定怎么打。

### 第一层：免费稳定可获取（Phase 0.5 可用）

| 数据类型 | 具体内容 | 获取方式 | 频率 | 延迟 |
|---|---|---|---|---|
| **日线交易数据** | 开高低收量、涨跌幅、换手率 | AkShare / Tushare | 每日 | T日收盘后 |
| **板块成分数据** | 板块树结构、成分股列表、板块指数 | 东方财富爬虫 | 每日 | T日收盘后 |

|---|---|---|---|---|

| **研报标题+摘要** | 东方财富"机构观点"、慧博公开研报 | 爬虫 | 免费但反爬严格 | 全文需付费，信息量有限 |

### 第三层：高噪音或不稳定（谨慎使用）

| 数据类型 | 具体内容 | 问题 |
|---|---|---|

| 工具 | 用途 | 说明 |
|---|---|---|

| **规则引擎** | 技术面评分、赔率检查、市场确认 | 纯代码实现，可回测 |
| **人工覆盖** | 修正错误、补充知识 | 可选，非必需 |

## 1. 重新审视：逻辑强度应该靠什么判断？

### 原始问题
新闻统计做逻辑强度 = 滞后 + 噪音。那逻辑强度到底靠什么？

### 方案对比

| 方案 | 优点 | 缺点 | 可用性 |
|---|---|---|---|
| **A. 新闻/研报数量统计** | 简单可算 | 滞后30%+涨幅后才密集；板块间不可比；可能是机构出货配合放消息 | ❌ 不可用 |
| **B. 用市场行为数据替代**（涨停家数、换手率变化） | 真实反映资金行为，每日稳定获取 | 本质是用价格解释价格，丧失了逻辑层的独立判断价值 | ⚠️ 可用于市场层，不适合逻辑层 |
| **C. LLM + Skill 锚定分类** | LLM 擅长信息提取和归类；Skill 提供锚点保证一致性；不依赖价格 | Skill 维护成本高；LLM 输出仍有波动；需要精心设计 Skill 才能避免拍脑袋 | ✅ 推荐，但需要投入 |
| **D. 产业事件驱动的规则引擎** | 政策发布=+1、业绩预增=+1、订单公告=+1，纯规则打分，稳定可回测 | 覆盖面有限，非标准化的事件难以规则化；初期规则库很小 | ✅ 推荐作为 Phase 0.5 方案 |
| **E. 盈利预期变化**（券商盈利预测上调比例） | 最接近"产业逻辑"的本质——盈利变化 | 券商预测本身有滞后和抱团；数据获取不稳定 | ⚠️ 可作为辅助 |
| **F. 用户手动输入** | 最准确（用户知道自己在关注什么逻辑） | 绝大多数用户不会维护，系统无法自动化 | ❌ 不可作为主方案 |

### 重新审视的结论

**推荐组合方案**：

```
逻辑强度 = 规则引擎打底（Phase 0.5）+ LLM+Skill 增强（Phase 1）

Phase 0.5（规则引擎）：
  政策事件计分板（政策发布/升级/落地/证伪）
  产业事件计分板（订单/涨价/业绩/产能）
  全球宏观因子（美联储/关税/大宗商品价格）

Phase 1（LLM+Skill）：
  LLM 提取新闻/研报中的"事件" → 映射到计分板
  Skill 提供各逻辑族的锚点标准
  输出等级分类（强/中/弱）而非绝对分数
```

**关键认知转变**：逻辑强度不应该是一个连续的 0-1 分数，而应该是**基于证据积累的等级判断**。就像信用评级（AAA/AA/A/BBB...），不是精确分数，而是区间等级。

## 2. 重新审视：周期/阶段判断真的可以放弃吗？

### 原始问题
用产业渗透率判断 early/mid/late/exhaustion = 频率错配。我们改为 `logic_heat_level`（新闻统计热度）。

### 方案对比

| 方案 | 优点 | 缺点 | 可用性 |
|---|---|---|---|
| **A. 产业渗透率/扩散度** | 理论上最准确 | 年频数据；个人开发者无法获取；与周频交易错配 | ❌ 不可用 |
| **B. 新闻统计热度**（当前方案） | 可计算 | 本质还是新闻数量；噪音大；热度高无法区分"加速扩散"vs"情绪过热" | ⚠️ 勉强可用 |
| **C. 市场状态间接推断** | 无需产业数据；基于已有数据；可计算 | 逻辑强度与价格的关系是事后的；可能有滞后 | ✅ 推荐 |
| **D. 完全放弃阶段判断** | 最简单；不做预测 | 丧失了"逻辑加速扩散"vs"逻辑衰竭"的区分能力 | ⚠️ 太保守 |

### 方案 C 的具体实现

**用"逻辑强度"与"市场确认"的关系来间接判断阶段**：

| 逻辑强度 | 市场确认 | 推断阶段 | 含义 |
|---|---|---|---|
| ↑ 增强 | ↑ 确认 | 加速期 | 逻辑被验证，市场开始交易 |
| ↑ 增强 | → 未变 | 左侧期 | 逻辑已强化但市场未反应，潜在机会 |
| → 稳定 | ↑ 确认 | 扩散期 | 逻辑不变，市场持续交易 |
| → 稳定 | ↓ 背离 | 疑似衰竭 | 逻辑未变但市场不交易了 |
| ↓ 减弱 | ↓ 背离 | 证伪期 | 逻辑被削弱，市场也不认可 |
| ↑ 增强 | ↓ 背离 | 分歧期 | 逻辑和市场出现巨大分歧，高分歧窗口 |

**这比新闻统计好在哪里？**
- 不需要额外的新闻爬取和分析
- 用的是已有的逻辑层和市场层输出，零额外数据成本
- 不是预测，而是对当前状态的描述
- 可以区分"加速扩散"和"情绪过热"（前者是逻辑+市场同向增强，后者是逻辑减弱但市场还在涨 = 纯炒作）

**缺点**：
- 是事后描述，不是领先判断
- 依赖于逻辑层评分的质量（如果逻辑层打分不准，这个推断也不准）

### 重新审视的结论

**放弃产业周期阶段判断是对的，但用新闻统计替代是错的。**

更好的方案是：**用逻辑强度与市场确认的关系来推断阶段**，这是零额外成本的"免费信号"。

## 3. 重新审视：LLM 到底能不能打分？

### 原始观点
"LLM 不应该做强度打分，只能做信息提取。"

### 自我质疑
这个说法太绝对了。问题是 LLM "自由打分"不稳定，不是"打分"本身不行。

### 方案对比

| 方案 | 优点 | 缺点 | 可用性 |
|---|---|---|---|
| **A. LLM 自由打分**（0-1连续） | 简单 | 一致性差，同一输入跑10次可能0.6-0.9 | ❌ 不可用 |
| **B. LLM 等级分类**（强/中/弱） | 比连续分数稳定；LLM 擅长分类 | 仍需 Skill 锚定；边界情况仍会波动 | ✅ 推荐 |
| **C. LLM + Skill 锚定打分** | Skill 提供明确的评分锚点；LLM 按锚点归类；可回溯到具体锚点 | Skill 维护成本高；需要为每种逻辑族设计 Skill | ✅ 推荐（长期） |
| **D. 纯规则引擎打分** | 完全稳定可回测；无 LLM 波动 | 覆盖面有限；非标准化事件难以规则化 | ✅ 推荐（Phase 0.5） |
| **E. LLM + DSPy 自动优化** | DSPy 自动调优 prompt；减少人工调参 | 技术栈复杂；需要标注数据做优化 | ⚠️ 远期可考虑 |
| **F. LLM 多采样取中位数** | 跑5-10次取中位数/众数；减少随机波动 | token 成本高；仍无法消除系统性偏见 | ⚠️ 可用但性价比低 |

### LLM + Skill 的具体设计

Skill 不是"让 LLM 自由打分"，而是"让 LLM 做锚点匹配"。示例：

```yaml
# skill: 政策驱动逻辑评估
dimensions:
  - name: "政策级别"
    anchors:
      - score: 1
        description: "部委层面表态，无具体措施"
        example: "某司长在论坛上提到要关注XX"
      - score: 3
        description: "国务院/部委发布指导意见，有方向但无时间表"
        example: "国务院印发XX发展指导意见"
      - score: 5
        description: "具体政策落地，有财政支持/时间表/考核指标"
        example: "XX补贴细则发布，明确补贴金额和截止时间"

  - name: "政策传导链条"
    anchors:
      - score: 1
        description: "传导链 ≥ 3 层，中间环节多"
        example: "碳中和 → 能源转型 → 光伏 → 组件 → 硅料"
      - score: 3
        description: "传导链 2 层，直接受益"
        example: "新能源汽车补贴 → 整车厂"
      - score: 5
        description: "传导链 1 层，直接受益，无中间环节"
        example: "降低 XX 关税 → 直接利好 XX 进出口"
```

LLM 的任务不是"这条逻辑打几分"，而是"这条新闻最匹配哪个锚点"。

### 重新审视的结论

**"LLM 不应该打分"这个说法需要修正**：

| 能力 | LLM 能做吗？ | 推荐方式 |
|---|---|---|
| 信息提取（提取事件、主体、关系） | ✅ 强 | 直接让 LLM 做 |
| 逻辑分类（事件驱动/产业趋势/政策驱动） | ✅ 强 | 直接让 LLM 做 |
| 自由打分（0-1 连续） | ❌ 差 | 不用 |
| 锚定等级分类（匹配 Skill 锚点） | ✅ 可行 | LLM + Skill 联合 |
| 生成解释（为什么给这个等级） | ✅ 强 | 让 LLM 引用具体锚点 |
| 定价判断（是否已被 price in） | ❌ 不能 | 交给市场层规则引擎 |

**关键区分**：不是"LLM 能不能打分"，而是"LLM 应该做自由打分还是锚定分类"。锚定分类 = 有参考系的归类判断，比自由打分稳定得多。

## 4. 重新审视：Skill 真的值得投入吗？

### 原始观点
Skill 可以锚定 LLM 评分，解决一致性问题。

### 自我质疑
Skill 维护成本可能远超预期。每个逻辑族都要设计 Skill，每个维度都要设计锚点，这本身就是一个巨大的工程。

### 投入产出分析

| 维度 | Skill 方案 | 纯规则方案 |
|---|---|---|
| **开发成本** | 高（需要为每种逻辑族设计 Skill，维护锚点） | 中（编写规则，但覆盖面有限） |
| **维护成本** | 高（行业经验变化需更新锚点） | 中（新增规则，修改阈值） |
| **可解释性** | 高（可回溯到具体锚点） | 高（规则透明） |
| **稳定性** | 中（仍有 LLM 波动，但比自由打分好） | 高（完全确定） |
| **覆盖广度** | 高（LLM 可处理非标准化事件） | 低（只能处理预定义的事件） |
| **回测可行性** | 中（LLM 输出有波动，回测结果可能有噪音） | 高（完全可复现） |

### 分阶段建议

| 阶段 | Skill 策略 |
|---|---|
| **Phase 0.5** | 不用 Skill。纯规则引擎（事件计分板）。先把最小闭环跑通。 |
| **Phase 1** | 为最常用的 1-2 种逻辑族设计 Skill（如"政策驱动"、"产业趋势"）。LLM 做锚定分类。 |
| **Phase 2** | 逐步扩展 Skill 覆盖更多逻辑族。建立 Skill 版本管理和回测验证流程。 |

### 投资理念的 Skill 设计方向

Skill 不仅是评分锚点，还可以融入经典投资理念作为"定性标签"：

| 理念 | Skill 维度 | 作用 |
|---|---|---|
| **巴菲特** | 护城河类型（品牌/成本/网络效应/转换成本/特许经营权） | 快速否决：无护城河的纯概念股降级 |
| **欧奈尔（CANSLIM）** | 业绩加速、相对强度、机构持仓变化 | 辅助判断个股是否能承接板块逻辑 |
| **利弗莫尔** | 关键价格位置（新高/新低）、市场趋势判断 | 市场层辅助：确认"市场在交易这个逻辑" |
| **索罗斯** | 反身性（逻辑是否自我强化） | 高分歧场景：判断是"分歧转一致"还是"逻辑证伪" |

这些 Skill 不直接参与量化计算，而是作为**定性标签和快速否决信号**。

### 重新审视的结论

**Skill 值得投入，但要分阶段**。Phase 0.5 不用，Phase 1 先做 1-2 个核心逻辑族的 Skill。投资理念 Skill 作为定性标签，不参与核心量化计算。

## 5. 重新审视：所有规则必须用日线数据计算

### 原始观点
"收盘后系统不碰盘中数据，所有规则必须能用日线数据计算。"

### 自我质疑
这个说法太绝对了。"收盘后可获取的数据"不等于"日线 OHLCV"。收盘后我们能拿到的数据远不止 OHLCV。

### 修正后的表述

**所有规则必须能用收盘后可获取的数据计算**，这包括：

| 数据类型 | 是否可用 | 说明 |
|---|---|---|
| 日线 OHLCV | ✅ | 最基础 |
| 涨跌停数据 | ✅ | 收盘后可获取 |
| 板块涨跌排名 | ✅ | 收盘后可获取 |
| 北向资金日度数据 | ✅ | 收盘后东方财富更新 |
| 新闻标题 | ✅ | 全天持续可获取 |
| 公告数据 | ✅ | 盘后密集发布 |
| 龙虎榜 | ✅ | 收盘后交易所公布 |
| 融资融券日度数据 | ✅ | 交易所每日公布 |
| 大宗交易 | ✅ | 收盘后可获取 |
| 盘中分时数据 | ⚠️ 谨慎 | 需要额外采集，但收盘后也可获取 |
| 实时竞价数据 | ❌ | 系统不涉及盘中执行 |

### 重新审视的结论

**"日线数据"这个说法太窄了。应该改为"收盘后可获取的数据"。** 这包括了日线 OHLCV、龙虎榜、北向资金、融资融券、公告等更丰富的数据。这大大扩展了系统设计空间。

## 6. 重新审视：宏观敏感度矩阵的维护成本

### 原始观点
"建立板块对宏观维度的敏感度矩阵，实现差异化加减分。"

### 自我质疑
几百个板块 × 5 个宏观维度 = 上千个敏感度参数。这个矩阵的维护成本太高了。

### 更好的方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 板块敏感度矩阵** | 最精确 | 维护成本极高（几百×5个参数） |
| **B. 逻辑族映射** | 不同逻辑族天然对宏观敏感度不同；只需维护 10 种逻辑族 × 5 维 | 精度下降，同一板块内不同逻辑敏感度不同 |
| **C. LLM 宏观解读** | LLM 自动输出受益/受损板块排序 | 一致性差；无法回测 |
| **D. 历史回归分析** | 用历史数据计算各板块对宏观指标的相关性 | 需要大量历史数据；相关性不等于因果关系 |

### 推荐方案

**Phase 0.5**：用逻辑族映射（方案 B）。例如：
- 产业趋势类逻辑 → 对"增长动能"敏感
- 流动性类逻辑 → 对"流动性环境"敏感
- 出口相关逻辑 → 对"全球联动"敏感

**Phase 2**：用历史回归分析（方案 D）校准逻辑族映射的参数。

### 重新审视的结论

**板块敏感度矩阵在 Phase 0.5 不现实。用逻辑族映射替代，精度够用，维护成本低。**

## 7. 重新审视：中军定义的动态混合方案

### 原始方案
中军 = 满足以下任意 2 项：市值前5 + 近20日日均成交额前5 + 用户手动标记

### 自我质疑
"前5名"在大盘股板块（如医药）和小盘股板块（如某个细分概念）含义完全不同。

### 更好的方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 绝对排名前5** | 简单 | 大盘股板块 vs 小盘股板块不可比 |
| **B. 板块内百分位** | 跨板块可比；自适应 | 需要定义百分位阈值 |
| **C. 板块内百分位 + 涨幅过滤** | 百分位确保地位；涨幅过滤排除长期下跌的"前中军" | 需要定义涨幅阈值 |
| **D. 市值百分位 + 成交额百分位 + 近N日涨幅** | 最全面 | 参数多 |

### 推荐方案

```
中军 = 满足以下条件：
  1. 市值在板块内 > 75 百分位（确保是"大票"）
  2. 近20日日均成交额在板块内 > 75 百分位（确保有流动性）
  3. 近60日涨跌幅不在板块后25%（排除长期下跌的"前中军"）

同时满足以上3项 → 中军
满足 1+2 但用户手动标记 → 中军
```

### 重新审视的结论

**用百分位替代绝对排名，加涨幅过滤排除长期下跌的"前中军"。**

## 8. 重新审视：分歧系数的必要性

### 原始设计
divergence_coefficient = min(pos, neg) / max(pos, neg)，>0.7 高分歧收紧止损

### 自我质疑
这个系数本身没有被回测验证过。在 Phase 0.5 阶段，可能根本不需要这么复杂的计算。

### 方案对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 线性净值得分** | 简单 | 消灭波动率信息 |
| **B. 分歧系数** | 保留波动率信息 | 参数未验证；计算复杂 |
| **C. 直接标记"高分歧"状态** | 简单；定性判断 | 需要定义高分歧的阈值 |
| **D. 正反逻辑数量比** | 极简；可解释 | 精度低 |

### 推荐方案

**Phase 0.5**：用方案 D（正反逻辑数量比）+ 方案 C（定性标记）。
```
if positive_count > 0 and negative_count > 0:
    标记为"分歧"
    执行层建议：收紧止损，降低仓位
```

**Phase 2**：引入分歧系数量化。

### 重新审视的结论

**Phase 0.5 不需要分歧系数。简单的"有无正反逻辑"标记就够了。**

## 9. 重新审视：趋势延续性检查（替代 T+1）

### 当前方案
首次出现的新逻辑 → 标记"观察股" → 建议等待 1-2 日趋势确认

### 自我质疑
这个方案在逻辑上是合理的，但"首次出现"的定义需要明确：是第一次被 LLM 识别到？还是第一次有新闻报道？还是用户第一次手动输入？

### 更好的方案

```
趋势延续性 = 该逻辑已被系统追踪的天数

首次出现（day 1）：
  → 标记为"新逻辑"，建议"观察"
  → 不否决，但建议轻仓

连续出现 2-3 天：
  → 标记为"验证中"
  → 正常执行

连续出现 >5 天：
  → 标记为"已确认"
  → 正常执行

这比"首次出现"的二元判断更平滑，且"被系统追踪的天数"是可精确计算的。
```

### 重新审视的结论

**用"逻辑被追踪天数"替代"首次出现"的二元判断。**

## 10. 修正后的方案汇总

| 原方案 | 问题 | 修正后方案 | 阶段 |
|---|---|---|---|
| 新闻统计做逻辑强度 | 滞后+噪音 | 规则引擎打底 + LLM+Skill 锚定分类 | Phase 0.5 / 1 |
| 新闻统计做热度判断周期 | 还是新闻统计 | 逻辑强度 × 市场确认关系推断阶段 | Phase 0.5 |
| LLM 不能打分 | 太绝对 | LLM 做锚定等级分类（强/中/弱），不做自由打分 | Phase 1 |
| Skill 全面铺开 | 维护成本太高 | Phase 0.5 不用，Phase 1 先做 1-2 个核心逻辑族 | Phase 0.5 / 1 |
| 日线数据计算 | 太窄 | 收盘后可获取的数据（含龙虎榜、北向、两融等） | Phase 0.5 |
| 板块敏感度矩阵 | 维护成本太高 | 逻辑族映射替代 | Phase 0.5 |
| 中军 = 前5名 | 跨板块不可比 | 百分位 + 涨幅过滤 | Phase 0.5 |
| 分歧系数 0.7/0.3 | 参数未验证 | Phase 0.5 只用"有无正反逻辑"标记 | Phase 0.5 / 2 |
| 首次出现二元判断 | 定义模糊 | 逻辑被追踪天数 | Phase 0.5 |

## 11. 核心架构的重新思考

基于以上审视，一个更务实的架构应该是：

```
┌─────────────────────────────────────────────────┐
│  逻辑层（不碰价格）                                │
│  输入：新闻标题、公告、政策文件、产业事件、宏观数据  │
│  处理：                                          │
│    Phase 0.5：规则引擎事件计分板                   │
│    Phase 1+：LLM + Skill 锚定等级分类             │
│  输出：                                          │
│    - 逻辑摘要、分类、角色                           │
│    - 逻辑等级（强/中/弱）而非精确分数               │
│    - 逻辑被追踪天数                                │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  市场层（纯规则引擎）                               │
│  输入：日线 OHLCV、板块行情、北向资金、龙虎榜        │
│  处理：                                          │
│    - 技术面评分（MA排列、量比、ATR）                │
│    - 资金面评分（北向近似、龙虎榜）                  │
│    - 情绪面评分（涨停家数、换手率变化）              │
│    - 赔率检查（涨幅、估值分位）                      │
│  输出：                                          │
│    - 市场确认状态（弱/中/强）                       │
│    - 市场-逻辑关系推断的阶段                        │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  执行层（规则引擎）                                 │
│  输入：逻辑层 + 市场层 + 宏观层                    │
│  处理：                                          │
│    - 逻辑×市场×宏观 决策矩阵                       │
│    - 趋势延续性检查（追踪天数）                      │
│    - 获利盘厚度检查                                │
│    - 涨跌停约束                                    │
│  输出：                                          │
│    - 方向 + 仓位 + 止损条件 + 等待条件              │
│    - 人话解释                                      │
└─────────────────────────────────────────────────┘
```
