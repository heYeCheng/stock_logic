# 交易逻辑选股系统设计避坑指南（幻方量化视角修正版）

> **Version**: 2.0-deepseek  
> **Date**: 2026-04-18  
> **适用场景**: A股逻辑驱动波段选股系统设计  
> **核心理念**: 数据驱动、系统化工程、AI与规则协作、务实迭代

---

## 前言：幻方量化的核心原则

作为 DeepSeek 的母公司，幻方量化在量化投资和 AI 应用上有几点核心理念，这些理念贯穿本文档：

1. **数据先行**：先确认能稳定获取什么数据，再设计模型。不要为不可靠的数据源设计复杂逻辑。
2. **简单有效优于复杂炫技**：能用线性模型解释的不用深度学习，能用规则表达的不强行用 AI。
3. **回测不是万能的**：A 股历史短、制度变化频繁，过拟合是最大的敌人。样本外验证和实盘小跑才是真正的检验。
4. **AI 做它擅长的，规则做它擅长的**：AI 擅长非结构化信息提取、模式识别、模糊匹配；规则擅长确定性计算、风控、执行纪律。
5. **系统永远有盲区**：设计时就要考虑降级和容错，而不是假设所有模块都完美运行。

## 一、数据源幻觉类

### 坑1：假设 Level-2 / 付费数据可获取

**严重度**：🔴 致命

**幻方实践**：在幻方的量化体系中，另类数据（卫星图像、供应链物流、电商评论等）确实被广泛探索，但有两个严格前提：
1. **覆盖度验证**：数据必须覆盖足够样本（如至少覆盖全市场 80% 的标的），否则无法用于系统性策略。
2. **alpha 衰减评估**：任何新数据源的 alpha 都会随使用人数增加而迅速衰减，必须有持续发现新数据源的能力。

**对于个人/小团队开发者**：以上两条几乎无法满足。因此，核心策略必须建立在**公开、稳定、低成本**的数据源上。

#### 正确做法及严谨评估

| 原方案（替代数据） | 幻方视角下的评估 |
|---|---|
| VWAP 乖离率替代筹码分布 | **可用但信息量极低**。日线 VWAP 仅是当日成交均价，完全无法刻画筹码在价格轴上的分布结构。可以用，但不要赋予过高权重。 |
| 换手率比率替代搜索热度 | **方向正确但需配合方向**。换手率是市场关注度的有效代理，但需结合价格方向（涨时放量 vs 跌时放量）才有意义。 |
| 北向 top10 替代全量 | **仅作情绪参考**。top10 覆盖市值大但数量少，不能代表板块全貌。 |

**幻方建议的数据源优先级**：
- **Tier 1（核心依赖）**：日线 OHLCV、涨跌停数据、板块成分股列表、财报基础数据（PE/PB/营收）。
- **Tier 2（辅助验证）**：北向资金日度汇总、龙虎榜、融资融券余额、公告标题。
- **Tier 3（探索性）**：新闻标题情感、社交媒体热度。这些只用于生成描述性文本，不参与量化决策。


### 坑2：把新闻关键词统计当作逻辑强度

**严重度**：🔴 致命

**幻方实践**：幻方在 NLP 上的投入巨大，但主要用于**实体识别、关系抽取、事件检测**，而非简单的词频统计。例如，从公告中提取"合同金额"、"中标项目"、"产能扩张计划"等结构化字段，这些才是可靠的事件信号。

**修正方案**：
- ❌ **不做**：统计新闻数量或关键词频率作为强度分数。
- ✅ **做**：用 LLM 进行**事件提取**（Event Extraction），输出结构化的事件记录。例如：
  ```json
  {
    "event_type": "policy_release",
    "entity": "工信部",
    "target_sector": "新能源车",
    "sentiment": "positive",
    "summary": "延续购置税减免政策至2027年"
  }
  ```
- 逻辑强度由**事件计分板**（Event Scoreboard）规则引擎计算：例如，一条"国家级政策利好"计 +0.2 分，一条"龙头公司业绩超预期"计 +0.15 分，累计并做时间衰减。这样既利用了 LLM 的信息提取能力，又保证了评分的稳定性和可解释性。


### 坑3：依赖研报作为领先指标

**严重度**：🔴 致命

**幻方实践**：卖方研报在量化模型中通常被视为**反向指标或噪音**。研报发布时信息已充分扩散，超额收益为负。幻方更关注**另类数据的前瞻性**，例如供应链订单、招聘数据、交通运输数据等。

**修正方案**：
- 研报的唯一用途：**确认市场共识**。当某个逻辑被多家研报密集覆盖时，意味着该逻辑已成为市场共识，此时应触发**赔率检查**（是否已充分定价）。
- 赔率检查的量化规则：当 `研报覆盖密度 > 阈值` 且 `板块近20日涨幅 > 30%` 时，自动将新开仓建议降级或否决。


### 坑4：用主力资金净流入作为核心因子

**严重度**：🟡 中等

**幻方实践**：幻方这类高频量化机构确实会分析逐笔成交数据，但目的是**拆解订单流、识别大单意图**，而非简单看净流入。对普通开发者而言，公开的"主力净流入"是高度加工后的低信噪比指标。

**修正方案**：
- 完全放弃"主力净流入"指标。
- 替代方案：**价格与成交量的配合关系**（价涨量增、价跌量缩等）是更原始、更难造假的信息。例如，计算"上涨日平均成交量 / 下跌日平均成交量"比率，能更好地反映资金倾向。


## 二、逻辑与价格因果倒置类

### 坑5：用价格涨跌反推逻辑强弱

**严重度**：🔴 致命

**幻方实践**：在幻方的多因子模型中，"逻辑因子"（如事件驱动、分析师预期）和"市场因子"（如动量、反转）是严格分离的两类因子，各自独立计算后再进行组合优化。绝不会用动量因子的值去修正逻辑因子的值。

**修正方案**：
- **逻辑层**：只基于**非价格事件**（政策、财报、公告、宏观数据）更新状态。维护一个事件日历，每天检查是否有新事件触发。
- **市场层**：只基于**量价数据**计算市场状态。
- 当价格大跌而逻辑层无负面事件时，系统应输出："逻辑层面无变化，市场层面出现抛压，建议观察支撑位，暂不否定逻辑。"这正是**避免情绪化交易**的关键价值。


### 坑6：让 LLM 同时承担逻辑发现和定价判断

**严重度**：🔴 致命

**幻方实践**：幻方对 LLM 的应用有清晰边界——用于**信息处理和知识表示**，而非**投资决策**。DeepSeek 可以阅读研报并总结核心观点，但不会让它直接输出"买入/卖出"指令。

**修正方案**：
- LLM 职责：从非结构化文本中提取**逻辑要素**（驱动因素、受益标的、风险点），并归类到预定义的逻辑类型中。
- 定价判断职责：完全由规则引擎基于量价数据完成（如计算相对强度、估值分位数、趋势位置）。
- 执行层：采用**决策矩阵**（逻辑等级 × 市场分数 × 赔率状态），而非端到端的黑盒模型。


### 坑7：LLM 输出的稳定性问题

**严重度**：🟡 中等

**幻方实践**：在生产环境中，LLM 的温度参数（temperature）通常设为 0 或极低值，以最大化输出的确定性。对于需要稳定性的任务，会采用 **few-shot prompting + 自一致性检查（Self-Consistency）**，即生成多个结果后投票。

**修正方案**：
- 对于逻辑强度打分，不依赖 LLM 直接输出分数。
- 采用 **LLM + Skill 锚定**：LLM 的任务是根据 Skill 中定义的锚点描述，将当前状态**归类**到某一档（如：业绩兑现度 = "高"），然后由规则引擎将"高"映射为具体的分数（如 0.8）。
- 这种方法将 LLM 的职责从"量化"转变为"分类"，大幅提升稳定性。


## 三、频率错配与信号滞后类

### 坑8：用慢变量定义快周期

**严重度**：🔴 致命

**幻方实践**：幻方在因子设计上严格区分**长周期因子**（如基本面价值）和**短周期因子**（如动量、流动性）。长周期因子用于选股池和基础仓位，短周期因子用于择时和交易执行。绝不会用季度财报数据指导日内交易。

**修正方案**：
- **放弃周期顶部预测**。不要试图判断逻辑处于 early/mid/late 哪个阶段，这需要产业高频数据，个人开发者无法获取。
- **改为状态监测**：
  - 逻辑是否仍然成立？（检查事件计分板是否有负面事件）
  - 市场是否还在交易？（检查技术面是否破位）
  - 赔率是否还合适？（检查涨幅和估值是否透支）
- 卖出信号由市场层触发（如跌破关键均线、量价背离），而非逻辑层预测。


## 四、模型过度复杂与过拟合类

### 坑9：多逻辑并发时设计过多角色权重维度

**严重度**：🟡 中等

**幻方实践**：幻方的多因子模型包含成百上千个因子，但每个因子的定义都非常简单清晰（如"过去20日收益率"）。复杂的因子通常样本外表现不佳。在逻辑层面，**简单加权求和往往比复杂公式更稳健**。

**修正方案**：
- Phase 1：正向逻辑最多保留 1 条主逻辑、1 条辅助逻辑；反向逻辑最多 1 条。权重分配：主逻辑 0.7，辅助 0.3。
- 净推力 = 主逻辑强度 × 0.7 + 辅助逻辑强度 × 0.3 - 反向逻辑强度。
- **不要引入**因果中心性、覆盖度、持续性等多维打分，除非你有稳定的数据源来客观计算这些维度。


### 坑10：过拟合于历史数据

**严重度**：🔴 致命（新增）

**幻方实践**：A 股历史数据最大的问题是**制度变迁**（注册制、涨跌停改革、融券新规等），导致历史规律频繁失效。幻方的做法是：
1. **缩短回测窗口**：重点看近 2-3 年数据，更早的权重降低。
2. **样本外验证**：至少保留最近 6 个月数据完全不用，用于最终验证。
3. **敏感性测试**：关键参数（如阈值）上下浮动 20%，看策略表现是否剧烈变化。如果剧烈变化，说明过拟合。

**修正方案**：
- 在需求文档中，所有阈值（如 0.3/0.5/0.8）都应标注为**可配置参数**，并留有接口供后续回测调优。
- 初期设计时，参数数量越少越好。每增加一个参数，就增加一分过拟合风险。


## 五、系统架构与工程落地类

### 坑11：收盘后系统混入盘中执行规则

**严重度**：🔴 致命

**幻方实践**：幻方的投研流程严格区分**盘后研究**和**盘中执行**。盘后系统负责选股池、信号生成；盘中系统只负责执行和风控，不重新计算核心信号。

**修正方案**：
- 选股系统的所有输入必须是在 **T 日 15:00 后**能够获取的数据。
- 可获取数据清单（扩展版）：
  - ✅ 日线 OHLCV、涨停板列表、龙虎榜、大宗交易、融资融券余额、公告、财报
  - ❌ 盘中分时数据、实时资金流向、Level-2 逐笔数据
- 铁律：**任何需要在盘中才能计算或确认的信号，不属于本选股系统**。


### 坑12：缺乏降级与容错设计

**严重度**：🟡 中等（新增）

**幻方实践**：在生产系统中，任何一个数据源或服务都可能挂掉。系统必须有降级方案。例如，如果新闻爬虫挂了，逻辑层就沿用前一天的逻辑状态，并标记"数据延迟"，同时自动降低仓位建议一档。

**修正方案**：
- 每个模块都要考虑其依赖的数据源不可用时的 fallback 逻辑。
- 例如：
  - LLM API 超时 → 使用上一日逻辑摘要，`logic_strength` 自然衰减 0.05。
  - 北向资金数据缺失 → 资金面评分置为 `None`，综合分由技术和情绪二维计算。
  - 财报数据缺失 → 跳过估值检查，不因此否决。


## 六、人工干预与协作边界

### 坑13：要求用户手动维护龙头列表

**严重度**：🟡 中等

**幻方实践**：幻方的系统高度自动化，但研究员和基金经理的**主动覆盖**仍然是重要环节。关键区别在于：机器负责覆盖 90% 的常规情况，人工只处理机器搞不定的 10% 边界案例，且人工覆盖会被记录并用于后续模型优化。

**修正方案**：
- **自动识别为主**：用规则（市值+成交额+涨幅）自动判定龙头和中军。
- **人工覆盖为辅**：提供界面让用户可以对任意个股标记"强制龙头"或"忽略"，这个标记会覆盖自动判断。**但这不是日常维护**，而是纠错机制。
- **反馈闭环**：用户的人工标记应被记录下来，定期分析，如果某种规则被频繁覆盖，说明规则需要优化。


### 坑14：让用户手动输入逻辑评分

**严重度**：🔴 致命

**幻方实践**：绝不让用户做系统能做的事。如果系统生成的逻辑不符合用户认知，用户可以不采纳系统的建议，但不需要用户每天帮系统纠错。

**修正方案**：
- 系统全自动生成逻辑摘要和强度。
- 用户唯一需要做的：**看或者不看，信或者不信**。
- 如果用户长期不信，说明系统失效，需要从模型层面优化，而不是靠用户输入来维持。


## 七、宏观层应用

### 坑15：宏观敏感度矩阵过于复杂

**严重度**：🟡 中等

**幻方实践**：幻方的宏观因子模型会考虑不同板块对利率、通胀、经济增长的敏感度（Beta），但这些 Beta 是通过**滚动回归**动态计算的，而非静态预设。

**修正方案**：
- Phase 0.5 **不做板块级别的宏观映射**。宏观仅作为一个全局加减分项（-0.1 ~ +0.1）。
- 若想做更精细的映射，可采用**逻辑族映射**替代板块映射：
  - "流动性驱动"逻辑（如券商）：对利率敏感度 +1。
  - "出口导向"逻辑（如家电）：对汇率敏感度 +1。
  - 这比维护几百个板块的敏感度表可行得多。


## 八、设计自检清单（幻方增强版）

1. **这个数据源在未来 6 个月还能稳定获取吗？**
2. **这个指标的计算逻辑在回测和实盘中完全一致吗？**（是否有未来函数）
3. **如果这个模块挂了，系统还能输出什么？**（降级方案）
4. **这个阈值如果改 20%，策略表现会天翻地覆吗？**（过拟合检测）
5. **用户需要每天做什么？**（如果 >0 件事，设计不合格）
6. **LLM 的输出是否被用于需要稳定性的计算？**（如果是，重新设计）
7. **这个规则在牛熊市中的表现会对称吗？**（A 股牛短熊长，很多规则在熊市失效）
8. **我是在预测未来，还是在描述当下？**（前者是圣杯，后者是系统）


## 九、核心铁律（最终版）

> **1. 数据先行，有多少米做多少饭。**  
> **2. 价格是最终裁判，事件是逻辑锚点，两者分层独立。**  
> **3. AI 做分类和提取，规则做计算和风控，各司其职。**  
> **4. 简单规则跑通前，不加任何复杂模块。**  
> **5. 每个模块都设计降级方案，系统永远可运行。**  
> **6. 用户是决策者，系统是信息处理助手，不要越俎代庖。**

## 总结：从设计到实盘的全链路自检清单

### 数据层面
- [ ] 复权方式是否全链路统一？
- [ ] 停牌、退市、ST是否正确处理？
- [ ] 涨跌停约束是否在回测中模拟？
- [ ] 财报、宏观数据的发布日期是否准确建模？

### 模型层面
- [ ] 是否做了严格的样本外验证（滚动窗口）？
- [ ] 参数数量是否合理？是否做了稳健性检验？
- [ ] 机器学习模型是否应用了正则化？
- [ ] 因子是否监控拥挤度和衰减趋势？

### 实盘层面
- [ ] 是否估算了策略容量和冲击成本？
- [ ] 是否有熔断和异常监控机制？
- [ ] 是否经过了纸面交易阶段验证？
- [ ] 回测与交易代码是否同源？

### LLM使用层面
- [ ] LLM是否直接输出交易决策？（绝对禁止）
- [ ] LLM输出是否有置信度过滤和人工抽样？
- [ ] 是否有规则兜底方案？
- [ ] Skill锚定是否经过一致性测试？

### 系统工程层面
- [ ] 数据源是否有冗余？
- [ ] 信号计算是否容忍数据延迟？
- [ ] 每日是否有自动对账流程？
- [ ] 运维告警是否覆盖关键指标？


## 新增章节一：量化策略全生命周期陷阱

### 坑20：历史回测完美 ≠ 实盘能赚钱

**严重度**：🔴 致命（量化第一杀手）

**错误做法**：
- 回测曲线漂亮（年化30%+，最大回撤<10%）就认为策略有效
- 用全样本数据优化参数，然后报告"样本内"表现
- 不考虑交易成本、冲击成本、涨跌停不可交易等现实约束

**为什么是坑**：
- **过拟合**：参数越多，越容易"拟合历史噪音"。一个带10个参数的策略，总能找到一组参数让回测曲线完美——但这组参数对未来毫无预测能力。
- **未来函数**：不知不觉使用了"当时还不知道"的信息。例如用"当日涨停"筛选但实际买入只能在次日，次日可能一字板买不到。
- **幸存者偏差**：回测池包含了后来退市的股票，但实盘时这些股票在退市前可能已被买入。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **严格的样本外验证** | 用滚动时间窗口：用过去N年数据训练/调参，用接下来M个月数据验证，滚动前进。幻方内部要求至少3轮完整的牛熊周期验证 |
| **交易成本建模** | 佣金+印花税+滑点（至少0.1%）+ 冲击成本（按成交额比例估算）。A股小盘股冲击成本可能高达0.3%-0.5% |
| **涨跌停可交易性检查** | 回测中，如果信号触发当日涨停，必须假设"买不到"，次日再判断 |
| **多套参数交叉验证** | 不是找"最好"的那组参数，而是找"参数稳健域"——参数微小变化时，表现不剧烈波动的区域 |
| **拒绝过度优化** | 幻方内部铁律：参数数量 ≤ 样本量的平方根。例如5年数据（约1200个交易日），参数不超过35个 |

**自检清单**：
- [ ] 回测是否使用了样本外数据？
- [ ] 是否考虑了T+1和涨跌停约束？
- [ ] 是否扣除了交易成本和滑点？
- [ ] 换手率是否合理？（日换手>50%的策略实盘无法执行）
- [ ] 参数是否进行了稳健性检验？


### 坑21：忽略了策略容量的天花板

**严重度**：🟡 中等（小资金无所谓，大资金致命）

**错误做法**：
- 回测时假设可以无限资金按收盘价成交
- 小盘股策略，回测表现优异，实盘上规模后收益骤降

**为什么是坑**：
- A股小盘股日均成交额可能只有几千万，一个500万的账户买入就会显著推高价格。
- 策略的"理论容量" = 标的日均成交额 × 可参与比例（通常<5%）。
- 冲击成本与订单规模呈非线性关系：买100万可能滑点0.1%，买500万可能滑点0.5%。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **容量估算** | 每个策略上线前必须估算最大容量 = min(成分股日均成交额 × 5%)。超过容量上限的资金，要么分散到更多标的，要么降低单票权重 |
| **分层执行** | 大订单拆分成多个小订单，用算法交易（TWAP/VWAP）执行，减少冲击 |
| **流动性过滤** | 日均成交额<5000万的标的，即使信号再强，大资金也不参与或极小仓位参与 |
| **冲击成本回测** | 高级回测框架会模拟订单簿，估算每笔交易的冲击成本。幻方内部有专门的"冲击成本模型" |


### 坑22：信号衰减比想象中快得多

**严重度**：🔴 致命

**错误做法**：
- 发现一个有效因子，高兴地部署上线，半年后发现失效了
- 不做信号衰减监控，策略表现下滑后才被动应对

**为什么是坑**：
- A股市场有效性在快速提升，任何公开的alpha因子都会迅速被市场消化。
- 因子拥挤：当越来越多资金使用类似策略时，信号领先性消失，甚至变成反向指标。
- 监管政策、市场微观结构变化（如涨跌停制度改革）也会导致因子失效。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **因子衰减监控** | 每个上线因子都有"生命周期仪表盘"：跟踪IC（信息系数）、IR（信息比率）、换手率、多空收益差等指标。任何指标连续3个月低于阈值，触发因子复审 |
| **因子库迭代** | 幻方维持数百个候选因子池，每月淘汰表现最差的10%-20%，同时从研究中补充新因子 |
| **多因子合成** | 单一因子容易失效，但多个低相关因子的组合往往更稳定。合成时注意因子共线性 |
| **市场状态自适应** | 不同市场环境下因子表现不同。例如动量因子在趋势市中有效，震荡市中失效。高级模型会根据市场状态动态调整因子权重 |
| **拒绝"圣杯"心态** | 没有永远有效的单因子。持续研究、持续迭代是唯一出路 |


### 坑23：实盘部署与回测环境不一致

**严重度**：🔴 致命

**错误做法**：
- 回测用Python/Pandas写的策略，直接翻译成实盘代码，但两者计算结果有微小差异
- 回测用的复权数据与实盘不一致
- 回测假设"次日开盘价成交"，实盘却是"次日9:30的成交价"

**为什么是坑**：
- 复权方式差异：回测用后复权，实盘用前复权，计算出的技术指标可能不同。
- 时间戳对齐：回测假设收盘后立即计算信号，实盘可能因为数据延迟，计算时间晚于收盘。
- 涨跌停处理：回测中"触发信号当日涨停"，实盘可能要到第三天才能买入。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **回测与交易代码同源** | 核心计算逻辑用同一套代码（或同一套DLL），杜绝翻译误差。幻方使用统一的C++计算引擎，Python只做调用 |
| **数据版本锁定** | 回测使用的数据版本必须与实盘数据源一致。每次数据供应商更新，必须重新验证回测结果 |
| **模拟撮合** | 实盘前必须经过"纸面交易"阶段——用真实行情模拟交易，验证信号计算、订单生成、成交假设是否与预期一致 |
| **可交易性检查前置** | 在回测阶段就模拟实盘约束：T+1、涨跌停、停牌、最小交易单位等 |
| **延迟容忍度测试** | 如果信号依赖收盘价，需要测试"晚1分钟拿到数据"对结果的影响。如果影响显著，说明策略太脆弱 |


## 新增章节二：LLM在金融场景中的特殊陷阱

### 坑24：用LLM直接生成交易信号

**严重度**：🔴 致命

**错误做法**：
- 让LLM阅读新闻后输出"买入/持有/卖出"建议
- 用LLM判断市场情绪（看多/看空），并据此交易

**为什么是坑**：
- LLM没有经过金融领域的专业微调，对"利好/利空"的判断与市场实际反应经常相反（例如"利好出尽是利空"）。
- LLM的知识截止日期通常滞后，无法获知最新市场共识。
- LLM存在"幻觉"，可能编造不存在的公告或数据。
- 即使微调过的模型，在金融预测任务上的表现也不稳定。

**幻方/DeepSeek实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **LLM只做信息提取，不做决策** | LLM从新闻/研报中提取结构化字段：事件类型、涉及公司、利好方向定性、影响期限预估。交易决策由规则引擎或量化模型完成 |
| **使用金融领域微调模型** | 通用LLM在金融文本理解上不如BloombergGPT、FinBERT等专业模型。幻方内部有专门的金融NLP团队做领域适配 |
| **置信度过滤** | LLM输出的每个字段都附带置信度，低置信度的结果丢弃或人工复核 |
| **集成规则兜底** | LLM失败时，系统自动降级为基于规则的文本匹配（关键词词典），保证系统可用性 |
| **永远不做"生成交易指令"** | 合规红线。LLM输出只用于"辅助分析"，不直接对接交易接口 |


### 坑25：忽视LLM推理的一致性问题（补充坑10）

**严重度**：🟡 中等

**补充内容**：即使使用了Skill锚定，LLM在以下场景仍然不稳定：

- **长文本理解**：当输入新闻超过8K token时，LLM容易丢失中间信息（"迷失在中间"现象）。
- **否定/反讽识别**：金融文本中大量存在"虽然...但是..."、"表面利好实则利空"的复杂语义，LLM容易误判。
- **数值推理**：让LLM比较"营收增速25% vs 市场预期30%"，它可能给出错误的大小判断。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **分块处理** | 长文档切成多个chunk分别提取信息，最后汇总，避免信息丢失 |
| **关键数值用正则提取** | 涉及数值比较的任务，先用正则提取数值，再用规则判断，不让LLM做数学 |
| **多轮验证** | 对重要判断（如业绩是否超预期），用不同prompt问两次，结果一致才采纳 |
| **人工抽样校验** | 每天抽样检查LLM输出质量，建立错误样本库，持续优化prompt和Skill |


## 新增章节三：数据工程陷阱

### 坑26：复权数据处理不当

**严重度**：🔴 致命（但新手极易犯错）

**错误做法**：
- 回测使用前复权，实盘使用不复权，导致技术指标计算结果不同
- 分红、送股时，未调整历史数据，导致收益率计算错误

**为什么是坑**：
- A股分红除权会导致价格跳空，如果不复权，技术指标（均线、布林带）会失真。
- 后复权保持历史价格不变，向后调整；前复权保持最新价格不变，向前调整。两者计算出的收益率不同。
- 实盘交易时看到的是"不复权价格"，但量化模型通常用"后复权"计算信号。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **统一使用后复权** | 全链路（数据存储→回测→信号计算）统一使用后复权价格。只在最终"展示"和"下单价格"时转回不复权 |
| **复权因子表维护** | 每日更新复权因子，确保任何时间点的回测都能正确复权 |
| **分红再投资假设** | 回测中明确是否考虑分红再投资。幻方默认假设"分红再投资"以反映真实收益 |
| **送股处理** | 送股会导致持仓数量变化，回测引擎必须正确处理 |


### 坑27：停牌/退市处理缺失

**严重度**：🔴 致命

**错误做法**：
- 回测中遇到停牌股票，假设可以正常交易
- 退市股票直接从回测池中删除，造成幸存者偏差

**为什么是坑**：
- 停牌期间无法买入也无法卖出，回测如果忽略这一点，会高估策略灵活性。
- 退市股票如果在退市前被买入，实盘会遭受巨大损失，但回测如果直接删除，就"美化"了业绩。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **停牌标记** | 每日维护停牌表。回测中，停牌股票不可买入；已持仓的停牌股票不可卖出，复牌后才能操作 |
| **退市股票保留** | 回测池保留所有历史存在过的股票（包括已退市），以消除幸存者偏差 |
| **ST处理** | ST股票的交易规则不同（涨跌幅5%），回测中需单独处理。幻方通常直接排除ST股 |
| **涨跌停处理** | 涨停时买不到，跌停时卖不掉。回测引擎需模拟这一约束 |


### 坑28：数据清洗的"度"

**严重度**：🟢 低（但容易被忽视）

**错误做法**：
- 过度清洗：把极端行情（如2015年股灾、2020年疫情暴跌）的数据当作"异常值"剔除
- 清洗不足：数据中的错误、缺失值未处理，导致指标计算出错

**为什么是坑**：
- 极端行情是市场的一部分，剔除后回测会高估策略的稳定性。
- 但如果不处理明显的错误数据（如某日价格突然为0），又会导致计算崩溃。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **只清洗明确错误** | 只剔除"价格≤0"、"成交量<0"、"涨跌幅超过理论限值"等明显错误。极端行情保留 |
| **缺失值前向填充** | 停牌期间的价格缺失，用前一日收盘价填充（保持持仓市值不变） |
| **异常波动标记** | 不删除异常波动，但标记出来。回测报告中单独分析"极端行情下的策略表现" |
| **数据质量仪表盘** | 每日监控数据源的完整性和准确性，数据质量下降时告警 |


## 新增章节四：模型与因子陷阱

### 坑29：因子拥挤与反身性

**严重度**：🟡 中等（量化策略的宿命）

**错误做法**：
- 发现某个因子有效，就不断加大权重
- 忽视市场上其他参与者也在使用类似策略

**为什么是坑**：
- 当大量资金使用同一因子时，因子信号本身就成为了价格推手，形成正反馈。一旦资金撤离，正反馈变成负反馈，踩踏出逃。
- A股近年来多次出现"量化策略踩踏"事件（如2024年初的小盘股流动性危机）。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **因子相关性监控** | 定期检查自营因子与市场公开因子的相关性。如果发现高度相关，说明策略可能已经拥挤 |
| **容量控制** | 每个因子策略有明确的规模上限，达到上限后不再新增资金 |
| **逆势保护** | 当因子信号与市场极端情绪反向时（如动量因子在恐慌性暴跌中发出买入信号），降低权重或暂停 |
| **分散化** | 不依赖单一因子。幻方同时运行数百个低相关性子策略，降低单一因子拥挤风险 |


### 坑30：过拟合的隐蔽形式

**严重度**：🔴 致命

**错误做法**（隐蔽形式）：
- 不断调整策略参数，直到回测曲线好看为止（"参数挖掘"）
- 用复杂的机器学习模型（XGBoost、神经网络）拟合历史数据，但不做正则化
- 样本外测试只有一次，如果那次恰好表现好，就认为模型有效

**为什么是坑**：
- 机器学习模型尤其容易过拟合，因为它们有大量参数。
- 金融数据的信噪比极低，模型很容易学到噪音而非信号。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **交叉验证** | 时间序列交叉验证：用滚动窗口，每轮用过去N个月训练，预测未来1个月。至少做10轮以上 |
| **正则化** | L1/L2正则化、Dropout、早停（early stopping）。幻方内部要求任何模型都必须有正则化项 |
| **特征数量限制** | 特征数量不超过样本量的1/10。例如1000个交易日的数据，最多用100个特征 |
| **样本外测试盲法** | 在最终评估前，样本外数据"封存"。只允许看一次结果，不允许根据结果回头调参 |
| **简单模型优先** | 能用线性回归解决的问题，不用神经网络。幻方大量策略仍然基于线性模型，因为更稳定、更可解释 |


## 新增章节五：系统工程与运维陷阱

### 坑31：数据延迟导致信号失效

**严重度**：🟡 中等（对高频策略致命，对日频策略影响较小）

**错误做法**：
- 回测假设收盘瞬间就能拿到所有数据并计算信号
- 实盘才发现，交易所数据有延迟，计算也需要时间，等信号出来已经收盘了

**为什么是坑**：
- 如果策略依赖收盘价（如"收盘站上MA20则买入"），实盘时你可能在收盘前几秒才能确认信号，来不及下单。
- 数据源（如东方财富、Wind）的收盘数据通常在15:05之后才完整推送。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **信号计算前置** | 尽量使用"盘中可计算"的指标。例如用14:55的价格近似收盘价，提前判断 |
| **延迟测试** | 回测中加入"数据延迟"模拟：假设信号基于T-1日数据生成，T日执行 |
| **收盘价替代** | 对于必须在收盘执行的策略，幻方采用TWAP算法在收盘前10分钟内均匀买入，减少对收盘价的冲击 |
| **数据源冗余** | 同时接入多个数据源（Wind、东方财富、交易所直连），一个延迟时自动切换 |


### 坑32：忽视策略监控与熔断机制

**严重度**：🔴 致命

**错误做法**：
- 策略上线后就不管了，直到某天发现亏了很多
- 没有设置最大回撤熔断，单日亏损超出预期仍在运行

**为什么是坑**：
- 市场环境可能突变（如2024年2月小盘股流动性危机），策略可能连续出错。
- 代码bug、数据错误也可能导致策略发出异常信号。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **实时监控仪表盘** | 监控策略的实时PnL、持仓集中度、因子暴露、换手率等。任何指标偏离历史均值2个标准差，触发告警 |
| **熔断机制** | 单日亏损超阈值（如-3%）暂停策略；连续3日亏损超阈值暂停；周度回撤超阈值暂停。熔断后需人工复核才能重启 |
| **订单异常检测** | 监控单笔订单金额是否异常（如突然比平时大10倍）、下单频率是否异常。防止"胖手指"或代码bug |
| **对账机制** | 每日收盘后，自动对比策略计算的持仓与券商实际持仓，不一致则告警 |


### 坑33：回测与实盘的时间戳对齐

**严重度**：🟡 中等

**错误做法**：
- 回测用自然日，实盘用交易日，导致信号日期错位
- 季度财报发布日期处理不当，回测假设财报发布当天就能拿到数据

**为什么是坑**：
- A股财报通常在收盘后发布，最早也要次日才能交易。回测如果假设发布当天就能交易，就会引入未来函数。
- 宏观数据（如PMI）通常在下个月第一天发布，回测如果假设当月最后一天就能拿到，也是错误的。

**幻方实践经验**：

| 正确做法 | 说明 |
|----------|------|
| **统一交易日历** | 全系统使用A股交易日历（剔除周末和节假日）。所有日期计算基于交易日 |
| **数据发布延迟建模** | 财报：假设T日收盘后发布，最早T+1日可交易。宏观数据：根据历史发布日历建模 |
| **公告时间精确到小时** | 如果数据源提供公告的具体时间（如"15:30发布"），回测中精确到小时，避免未来函数 |

**最后的话**：这份指南的每一个"正确做法"都代表一种**权衡**。没有完美的方案，只有当前约束下的最优解。在 A 股做逻辑选股系统，本质是在**信息不完备、数据高噪声、制度频繁变**的环境下寻求一个可用的概率优势。保持谦逊，持续迭代，先让系统跑起来，再逐步优化——这才是幻方式的工程哲学。
