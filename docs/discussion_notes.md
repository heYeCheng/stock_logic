# 需求设定讨论笔记

> 记录从 v3.0 到 v5.2-practical 迭代过程中确立的核心规则、规则背后的原因、以及反复踩过的坑。
> 供未来开发者、维护者、以及新加入的协作者参考。

---

## 1. 系统定位：这到底是什么

### 规则：这是一个"收盘后每日一次的波段选股系统"，不是择时系统，不是日内交易系统

**原因**：
- 系统使命是回答"市场在交易什么逻辑"、"逻辑还在不在"、"哪些股票最能承接这个逻辑"
- 解决的是散户的三个痛点：错过入场、被噪音震出、信息过载
- 波段交易（持仓天~周级别），不需要盘中实时信号

**坑**：
- 早期试图加入 T+1 盘中执行检查（竞价观察、开盘30分钟承接力度等），但这与"收盘后分析"的定位根本冲突
- **v5.2 最终修复**：将所有执行层约束替换为基于日线数据可计算的规则（获利盘厚度、趋势延续性、涨跌停约束）

---

## 2. 三层决策架构

### 规则：逻辑层、市场层、执行层必须严格分层，因果顺序不可逆

**架构**：
```
逻辑层（真不真？）→ 市场层（交易了吗？）→ 执行层（做不做？）
```

**原因**：
- 先用逻辑识别"市场在交易什么"，再用市场确认"有没有人跟进"，最后在执行层做"做不做、做多大、怎么退"的决策
- 用价格倒推逻辑 = 后视镜决策，会导致"逻辑评分很高但买进去就接盘"
- 用执行修改逻辑 = 循环论证，系统永远无法发现逻辑本身有问题

**反复踩的坑**：
| 坑 | 具体表现 | 最终方案 |
|---|---|---|
| **LLM 预训练污染** | LLM 知道最近光伏涨了，给光伏打高分本质是价格偏见的投影 | Prompt 级隔离（不输入价格数据）+ 人工标记"价格跟随型逻辑" |
| **研报滞后性** | 研报密集覆盖时股价通常已涨30%-50% | `logic_heat_level`（热度）+ `logic_pricing_saturation`（饱和度）联合判断 |
| **逻辑评分高 = 可买入？** | 否。高逻辑评分可能意味着已被充分定价 | 新开仓看赔率，持有看逻辑真假 |

---

## 3. 逻辑行为与市场行为的严格隔离

### 规则：逻辑评分不引用价格数据；市场雷达不反向修改逻辑评分

**允许输入/禁止输入矩阵**：

| 层 | 允许 | 禁止 |
|---|---|---|
| 逻辑层 | 政策、产业数据、财报、研报、供应链 | 价格涨跌、量价行为、热度 |
| 市场层 | 价格、量能、资金流、情绪 | 政策真值、产业真值 |
| 执行层 | 逻辑层输出 + 市场层输出 + 宏观 | 直接用未来收益修正当日动作 |

**工程约束（LLM 隔离的现实妥协）**：
1. **Prompt 级隔离**：输入给 LLM 的内容不包含任何价格、涨跌幅、成交量数据
2. **数据源软隔离**：隔离是设计意图，非物理定律。LLM 预训练已有价格认知偏见
3. **人工标记兜底**：发现逻辑评分与市场高度同步时，手动标记降低权重

**为什么不能完全隔离**：
- LLM 在预训练中已经见过所有历史价格，无法让它"忘掉"走势
- 研报本身是滞后确认信号，不是领先发现信号
- **所以**：隔离的目的是防止"把价格行为包装成逻辑证据"，而不是追求不可能的绝对纯净

---

## 4. role_weight 为什么只保留两维

### 规则：role_weight = f(causal_centrality, persistence)，不引入 coverage_breadth 和 irreplaceability

**原因**：
- causal_centrality 回答"这个逻辑是否解释了板块最核心的盈利/定价驱动"
- persistence 回答"这是一次性冲击还是可持续的主线"
- 两维已经能区分快/中/慢三套模板的权重分配
- coverage_breadth 和 irreplaceability 在实证中容易与 causal_centrality 高度相关，引入后边际信息量有限

**三套模板**：
| 逻辑族 | 公式 | 含义 |
|---|---|---|
| 快（事件驱动） | 0.6*causal + 0.4*persistence | 事件驱动更看重因果性 |
| 中（政策驱动） | 0.5*causal + 0.5*persistence | 平衡 |
| 慢（产业趋势） | 0.4*causal + 0.6*persistence | 产业趋势更看重持续性 |

**坑**：后续可以通过回测验证是否扩展，不要现在就把所有维度都塞进来。

---

## 5. 逻辑生命周期 → 逻辑状态追踪

### 规则：废弃 `evidence_cycle_stage`（early/mid/late/exhaustion），改用 `logic_heat_level` + `logic_trend`

**原因**：
- `early/mid/late/exhaustion` 需要高频产业数据，个人开发者无法获取
- 产业渗透率、扩散度是**年频变化**的慢变量，用来指导**周频交易**必然错位
- 典型后果："逻辑还在中期，股价已经腰斩"或"逻辑显示晚期，股价还在主升浪"
- `early` 阶段研报极少、噪音极大，LLM 无法区分"早期真逻辑"和"伪逻辑"
- 试图判断 `exhaustion`（衰竭）本身就是**强预测**，违反"跟随不预测"哲学

**替代方案**：
- `logic_heat_level`（0-10）：基于新闻/研报数量短期 vs 长期比率，描述讨论热度
- `logic_trend`：连续5日 `logic_strength` 的线性回归斜率，描述逻辑强度自身变化方向
- 热度 0-3=冷门，4-7=正常，8-10=过热（否决新开仓）

**关键原则**：系统不预测周期顶部，只提示"当前热度状态"。卖出靠市场确认（价格破位），不靠逻辑预测（"我觉得要结束了"）。

## 6. 多逻辑并发与信噪比

### 规则：一个板块可同时存在多条逻辑，用 SNR 和 divergence_coefficient 描述博弈状态

**叙事信噪比（SNR）**：
```
SNR = 主导逻辑得分 / (次要逻辑得分总和 + 反向逻辑得分)
```

红线规则：
- SNR < 1.5 → 强制"混沌"状态，空仓观察
- SNR 连续 3 日下降 → 标记"叙事模糊化"，减仓观察
- 牛市中动态阈值降至 1.0

**分歧系数（divergence_coefficient）**：
```
divergence_coefficient = min(positive_core, negative_core) / max(positive_core, negative_core)
```

**为什么不能用 positive - negative 线性相减**：
- 正向0.8 - 反向0.7 = 净值0.1 → 系统建议"轻仓/观察"
- 但 A 股中高分歧 = **巨大波动率**，不是温和观望
- 高分歧往往是**变盘前夜**或**最佳短线博弈窗口**
- **正确做法**：保留方向判断，收紧止损宽度（如10日线→5日线），建议仓位降一档

## 7. 市场确认状态机

### 规则：5 状态内部用连续 confirmation_score，离散状态仅展示

**状态**：unrecognized → validating → confirmed → crowded → invalidated

**为什么用连续系数**：
- 硬阈值会导致决策断裂（0.49 和 0.51 动作完全不同不合理）
- 内部用连续值做计算，离散标签仅用于 UI 展示

**为什么不用独立打分**：
- 市场确认状态应由雷达图 3 维阈值组合推导，不是独立变量
- 三维数据源必须互斥：技术面（均线/ATR/量比）、资金面（主力净流入）、情绪面（涨停家数/换手率变化）

---

## 8. 中军定义

### 规则：中军 = 满足以下任意 2 项

1. 市值位于该板块前5名（静态）
2. 近20日日均成交额位于该板块前5名（动态）
3. 用户手动标记的"行业龙头列表"中（静态）

**为什么不用研报标签单独定义**：
- 如果靠研报里的"行业龙头"标签，中国平安、万科A在下跌趋势中永远是中军
- 系统会陷入**永无止境的抄底陷阱**
- 用市值+成交额的动态混合定义，确保中军既是逻辑概念也是市场交易出来的结果

**交叉验证豁免**：
- 板块强 + 个股弱（但满足中军定义）→ 重点关注（豁免跳过）
- 板块强 + 个股弱（不满足中军定义）→ 跳过，找龙头

## 9. 执行层设计

### 规则：entry_readiness 按逻辑族差异化权重，快逻辑更依赖市场确认

**为什么快逻辑更依赖市场**：
- 快逻辑（事件驱动、制度变革）证据体系不稳定，需要更强的市场确认来弥补逻辑不确定
- 慢逻辑（产业趋势、供需周期）证据体系稳固，更依赖逻辑真值本身

**价格衰减因子**：
- 不在逻辑层引入价格，仅在执行层做性价比调整
- 快逻辑用 MA30 偏离，中逻辑用 MA120 偏离，慢逻辑用 PE/PB 历史分位数

**持有决策矩阵（区别于进入决策）**：
- 新开仓看赔率（saturation），持有看趋势（market_validation_state）
- 即使 saturation=high，只要趋势还在，不主动减仓，只收紧止损
- 1 项标记→减仓观察，2 项+→退出，逻辑失效/市场证伪→立即退出

## 10. T+1 规则的演变与最终放弃

### 演变路径：

| 版本 | 方案 | 问题 |
|---|---|---|
| v5.0 | 晚间新闻→等待收盘二次确认 | 重大利好次日一字板，等待已无意义；高开低走时确认时逻辑已破坏 |
| v5.1 | 晚间新闻→竞价观察清单 + 开盘30分钟提示 | 系统不涉及盘中执行，与定位冲突；需要用户看盘配合 |
| **v5.2** | **趋势延续性检查**：首次信号需1-2日确认 | **基于日线数据，与系统定位一致** |

**核心教训**：不要在收盘后分析的系统中设计盘中执行规则。所有约束必须能用日线数据计算。

## 11. 数据源降级

### 规则：个人开发者无法获取 Level-2、机构研报 API、实时舆情，必须降级

| 原需求 | 替代方案 | 原因 |
|---|---|---|
| 筹码密集区位移 | VWAP 与收盘价乖离率 | Level-2 数据，全推流重建每一笔成交 |
| 搜索指数变化率 | 换手率 3日/10日 比率 | 只有相对热度，T+1延迟，涨停当晚才飙升 |
| 北向资金持仓变动 | 东方财富公开北向净买入（仅沪港通top10） | 精度下降，仅作参考 |
| 研报摘要与评分 | 东方财富"机构观点"聚合页 | 仅统计"看好/看淡"数量 |
| 产业高频数据 | 手动输入 | 用户自填或预设日历 |

**坑**：不要在系统设计阶段假设"以后能拿到这些数据"。Phase 0.5 必须用实际能获取的数据跑通闭环。

---

## 12. 跟随不预测哲学的落地

### 规则：系统不做点位预测，不判断周期顶部，不预测逻辑何时结束

**具体表现**：
- ❌ "预计将涨到 XX 价位" — 不做
- ❌ "逻辑进入晚期，即将结束" — 不判断
- ❌ "目标价 XX 元" — 不做
- ❌ "开盘30分钟观察承接力度" — 盘中执行，不做

**替代方案**：
- ✅ "当前信号显示 XX，建议做 YY" — 观察-响应
- ✅ "strength declining，建议减仓" — 对当前状态的响应
- ✅ "若关键支撑区域失守则离场" — 条件触发

**卖出逻辑**：卖出靠市场确认（价格破位），不靠逻辑预测（"我觉得要结束了"）。逻辑只判断"还在不在"，"该不该走"由市场雷达图说了算。

## 13. 反复出现的三大坑

### 坑 1：用慢变量定义快周期

产业渗透率、研报密度是年频变化的指标，用来指导周频交易，必然错位。
**修复**：改为 `logic_heat_level`（信息热度变化率），描述讨论热度而非产业阶段。

### 坑 2：线性净值得分消灭波动率信息

positive - negative = 净值 → 高分歧被压缩为"温和观望"，消灭了变盘前夜的高赔率窗口。
**修复**：引入 `divergence_coefficient`，高分歧不否决但收紧风控。

### 坑 3：收盘后系统 vs 盘中执行

T+1 隔夜检查、竞价观察、开盘30分钟提示等规则，都要求用户在盘中执行判断，与系统定位根本冲突。
**修复**：所有约束改为基于日线数据可计算（获利盘厚度、趋势延续性、涨跌停）。

## 15. 尚未验证的假设

以下内容在文档中已确定，但**未经回测或实盘验证**，实现时应谨慎对待：

| 假设 | 当前取值 | 验证方式 |
|---|---|---|
| SNR 阈值 1.5 | 经验值 | 历史回测验证 |
| divergence_coefficient 0.7 高分歧阈值 | 经验值 | 历史回测验证 |
| logic_heat_level 8-10 过热否决 | 经验值 | 历史回测验证 |
| entry_readiness 权重分配 | 经验值 | 历史回测验证 |
| 价格衰减因子 MA30/MA120/MA200 基准 | 经验值 | 历史回测验证 |
| role_weight 两维公式 | 经验值 | 后续验证扩展 |
| 确认状态机阈值 0.3/0.5/0.8 | 经验值 | 历史回测验证 |
| SNR 连续3日下降标记模糊化 | 经验值 | 历史回测验证 |

**核心原则**：所有阈值都是起点，不是终点。Phase 3 回测阶段必须逐一验证和调整。

## 16. 给未来开发者的建议

1. **先跑通 Phase 0.5（最小闭环）**，不要急着上 LLM。手动输入逻辑评分 + 日线数据计算市场雷达图 + 简单排序，就已经能验证核心架构是否成立。

2. **LLM 逻辑评分不是必须的**。如果数据采集不稳定，LLM 输出质量会大幅下降。Phase 1 可以先只做用户自选股的 LLM 辅助分析，保留手动修正入口。

3. **所有阈值都应该是可配置的**。不要把 1.5、0.7、0.3 等数字硬编码到逻辑里。用 YAML 或 JSON 配置，方便回测调参。

4. **不要追求"完美的逻辑真值"**。A 股交易是信息不完备下的概率博弈，不是有标准答案的数学题。系统的价值在于减少情绪化交易，不在于给出"绝对正确"的建议。

5. **卖出比买入更容易定义**。买入需要预测赔率，卖出只需要确认失效条件。先把失效条件（逻辑断裂、评分塌陷、市场证伪）定义清楚，系统就已经有实用价值了。
